    1. Introducción.

    Git es una herramienta de control de versiones que facilita la administración de modificaciones en proyectos de software. Es clave para el trabajo en equipo, el seguimiento de versiones y la conservación del registro de cambios en el código.

    2. Configuración Inicial de Git.

    Configuración básica para usar Git.

    • config --global user.name "Tu Nombre": Este comando establece el nombre de usuario que se asociará a los commits.
    • Git config --global user.email "tuemail@ejemplo.com": Con este comando, asignas tu correo electrónico al usuario para que se registre en los cambios.
    • Git config --global core.editor "nombre_editor": Este comando define el editor de texto predeterminado que usarás en Git (es opcional).

    Estos comandos establecen tu nombre y correo electrónico en Git. Cada vez que hagas un commit, se asociará con esta información, lo cual es esencial para mantener un registro claro en el historial de cambios.

    3. Comandos Básicos de Git.

    Comandos esenciales para comenzar a trabajar con Git.

    • Git init: Crea un nuevo repositorio local en la carpeta actual.
    • Git status: Muestra el estado del repositorio, incluyendo archivos modificados, listos para commit o sin seguimiento.
    • Git add [archivo/patrón]: Añade un archivo o conjunto de archivos al área de staging, preparándolos para el commit.
    • Git add . : Añade todos los archivos modificados al área de staging.
    • Git commit -m 'mensaje': Guarda los cambios en el repositorio local con un mensaje que describe lo que se hizo.
    • Git commit --amend -m 'Nuevo mensaje': Permite cambiar el mensaje del último commit y hacer pequeños ajustes en él.
    • Git log: Muestra el historial completo de todos los commits con detalles.
    • Git log --oneline: Muestra el historial de commits de forma resumida, con un commit por línea.

    Estos comandos te permiten crear y administrar un repositorio de Git. El proceso básico consiste en añadir archivos, confirmar los cambios con un commit y consultar el historial de versiones.

    4. Comandos para Comparar y Revisar Cambios

    Herramientas para comparar versiones y ver las diferencias entre archivos o commits.

    • Git diff: Muestra las diferencias entre el contenido actual y los cambios que aún no has confirmado.
    • Git diff "commit1" "commit2": Compara dos commits específicos y muestra las diferencias entre ellos.

    Comparar archivos es fundamental para entender qué modificaciones se han realizado en el código antes de confirmarlas o entre versiones específicas.

    5. Gestión de Repositorios Remotos.

    ¿Cómo gestionar repositorios remotos para actualizar los cambios locales en un servidor?.

    • Git remote: Muestra todos los repositorios remotos que has configurado.
    • Git remote add [nombre] [URL]: Agrega un nuevo repositorio remoto utilizando un nombre y una URL específicos.
    • Git remote show [nombre]: Proporciona información detallada sobre un repositorio remoto específico.
    • Git remote rename [old] [new]: Cambia el nombre de una referencia a un repositorio remoto.
    • Git remote rm [nombre]: Elimina la referencia a un repositorio remoto de tu sistema local.

    Los repositorios remotos permiten trabajar de manera colaborativa, compartiendo el código a través de la nube o servidores como GitHub. Estos comandos hacen más fácil agregar, modificar y eliminar conexiones remotas.

    6. Subida de Cambios al Servidor Remoto.

    ¿Cómo envía los cambios que has realizado en tu equipo a un servidor remoto.? Los siguientes comandos te lo mostrarán.

    • Git push [servidor_remoto] [rama]: Este comando envía los cambios locales de la rama que especifiques al repositorio remoto.
    • Git push -u [repositorio_remoto] [rama_local]: Con este comando, estableces una conexión de subida (upstream) entre tu rama local y la remota, lo que hace más fácil realizar envíos en el futuro.
    • Git branch -M main: Este comando renombra la rama local "master" a "main", siguiendo la convención que usa GitHub.

    La sincronización con un servidor remoto permite que otros colaboradores accedan a los cambios que has realizado localmente. El comando "git push" se utiliza para enviar tus commits al servidor remoto.

    7. Rollback (Reversión de Cambios).

    ¿cómo deshacer o revertir cambios en el proyecto?.

    • Git reset HEAD [archivo]: Este comando devuelve el archivo al estado de no preparado (unstaged).
    • Git reset --soft HEAD^: Con este comando, se elimina el último commit, pero se conservan los cambios en el área de staging.
    • Git reset --hard [commit]: Este comando elimina todos los cambios y restablece el repositorio a un estado específico de un commit.
    • Git revert [HEAD^ || id_commit]: Crea un nuevo commit que deshace los cambios de un commit anterior, manteniendo así el historial intacto.

    Git ofrece diferentes maneras de deshacer cambios de forma segura. Aunque Git reset puede eliminar commits o modificaciones, Git revert es la opción más segura para deshacer cambios, ya que lo hace sin perder el historial.

    8. Manejo de Etiquetas en Git.

    ¿Cuál es el uso de etiquetas para marcar versiones específicas en el historial del proyecto?.

    • Git tag: Este comando muestra todas las etiquetas que existen en el repositorio.
    • Git show [nombre_etiqueta]: Con este comando, puedes ver detalles sobre una etiqueta, incluyendo el commit al que está vinculada.
    • Git tag -a [nombre_etiqueta] -m 'Mensaje': Este comando crea una etiqueta anotada en el commit actual y le añade un mensaje.
    • Git tag -a [id_commit] -m 'Mensaje': Este comando crea una etiqueta anotada en un commit específico y también le agrega un mensaje.

    Las etiquetas son útiles para señalar versiones clave del proyecto, como los lanzamientos y permiten tener una referencia rápida.

    9. Limpieza de Archivos y Conflictos.

    ¿Cómo eliminar archivos no deseados o gestionar conflictos en el repositorio?.

    • Git clean -n: Este comando muestra una lista de archivos que no están siendo rastreados (untracked) y que se eliminarían.
    • Git clean -f: Con este comando, se eliminan de forma permanente los archivos que no están siendo rastreados por Git.
    • Git checkout [id_commit]: Este comando te permite ver cómo estaba el proyecto en un commit específico, sin afectar el estado actual del proyecto.

    Estos comandos te permiten eliminar archivos que no necesitas y revisar versiones anteriores del proyecto sin cambiar el estado actual. Limpiar los archivos asegura que el repositorio no contenga archivos temporales o innecesarios.

    Fin del tema 1.

    Unidad 2. Creación y Gestión de Ramas.

    1.¿Qué son las ramas(Branch)?.

    Las ramas en Git son bifurcaciones que permiten desarrollar nuevas versiones de un proyecto sin modificar la rama principal, conocida como "master".

    • Trabajar con ramas en Git significa hacer cambios (commits) en una rama específica, lo que ayuda a mantener el trabajo organizado y evita interferencias entre diferentes desarrollos.
    • Al crear una nueva rama, se copia el estado actual del proyecto para seguir trabajando desde allí sin afectar la rama principal.

    2.Creación y Gestión de Ramas.

    A continuación explicaremos cómo listar, crear, eliminar y renombrar ramas en Git usando comandos.

    • Git branch: Muestra todas las ramas existentes y resalta la rama activa.
    • Git branch [nombre_nueva_rama]: Genera una nueva rama a partir de la actual.
    • Git branch -d [nombre_nueva_rama]: Borra la rama especificada, siempre que no tenga cambios no fusionados.
    • Git branch -m [old_nombre_rama] [new_nombre_rama]: Cambia el nombre de una rama existente.
    • Git show: Muestra información sobre el último commit y la rama asociada.

    La gestión de ramas es importante para organizar un proyecto. Los errores más comunes pueden ser eliminar ramas con cambios no fusionados o cometer errores al renombrarlas. Por eso, es importante seguir un procedimiento claro y revisar el estado de las ramas antes de hacer cambios.

    3. Historial de Commits.

    ¿Cómo ver el historial de cambios en el repositorio a través de la línea de comandos?.

    • Git log: Muestra una lista de los IDs de los commits con detalles.
    • Git log -p: Incluye cambios específicos por archivo en cada commit.
    • Git log --oneline: Muestra el historial en una línea por commit.
    • Git log --oneline --all: Muestra los commits de todas las ramas de forma compacta.

    Consultar el historial de commits es fundamental para entender los cambios en un proyecto. Con comandos como Git log podemos ver detalles de cada modificación. Un error común es no revisar este historial antes de hacer cambios importantes, no revisar todo el historial puede hacer que se pasen por alto commits relevantes, haciendo mas difícil detectar errores.

    4. Cambio entre Ramas.

    Vamos a explica cómo cambiar entre ramas, crear una rama nueva en un solo paso y regresar a una versión anterior del proyecto.

    • Git checkout [nombre_rama]: Cambia a la rama indicada.
    • Git checkout -b [nombre_nueva_rama]:Crea y cambia a una nueva rama simultáneamente.
    • Git checkout [ID_commit]: Permite volver a una versión anterior del proyecto especificando el ID del commit.

    Manejar ramas en Git es esencial para el flujo de trabajo en Git. Es importante guardar los cambios antes de cambiar de rama para evitar pérdidas de trabajo. Regresar a un commit anterior puede ser confuso si no se entiende el contexto, lo que puede generar errores. Por lo tanto, siempre asegúrate de guardar y comprender tus cambios antes de cambiar de rama.

    5. Comparación y Resolución de Conflictos.

    Veremos cómo visualizar diferencias y detectar conflictos entre ramas, lo cual es útil antes de realizar una fusión.

    • Git diff: Muestra diferencias entre el trabajo actual y los cambios pendientes.
    • Git diff [branch_origen] [branch_destino]: Compara dos ramas y muestra los posibles conflictos antes de fusionarlas.

    La visualización de diferencias nos permite entender qué cambios se han realizado y cómo pueden afectar el proyecto. Un error común es no realizar esta comparación antes de intentar fusionar, lo que puede provocar conflictos que dificultan el progreso del proyecto.

    6. Fusión de Ramas.

    Comandos para fusionar ramas y cómo manejar conflictos que puedan surgir durante este proceso.

    • Git merge [nombre_rama]: Se utiliza para fusionar los cambios de una rama específica en la rama en la que te encuentras actualmente. .
    • Git merge --abort: Permite revertir a la situación anterior si surgen problemas durante la fusión.
    • Git log --merge: Muestra información sobre los commits que causaron conflictos durante una fusión, útil para resolverlos.

    La fusión de ramas es un proceso que permite combinar cambios de diferentes líneas de desarrollo. Este proceso puede ser complicado si los conflictos que surgen no se manejan adecuadamente. Un error frecuente es no estar preparado para resolver conflictos, lo que puede llevar a que se implemente código que no funcione correctamente.

    Fin del tema 2.

    Unidad 3. Empezando a funcionar.

    1. Clonación de repositorios.

    Para comenzar a trabajar con un proyecto en GIT, primero debemos clonar el repositorio remoto a nuestro equipo local.

    • Git clone [remote_repo] [directory_local]: Clona un repositorio remoto en un directorio local.
    • Git clone --branch [tag] [remote_repo]: Clona una versión específica del repositorio.
    • Git clone --branch --single-branch [branch_name] [remote_repo]: Clona únicamente una rama específica del repositorio.

    Clonar el repositorio te permite trabajar en tu propio entorno de desarrollo sin alterar el repositorio original. Esto es crucial para colaborar en proyectos de software de manera eficiente.

    Sus errores comunes pueden ser no especificar correctamente la URL del repositorio o clonar todo el repositorio cuando solo se necesita una rama específica.

    2. Actualización de repositorios locales.

    Después de trabajar en local, es importante mantener sincronizados los cambios con el repositorio remoto.

    • Git pull [servidor_remoto]: Fusiona los cambios del remoto al local (fetch + merge).
    • Git pull [servidor_remoto] [rama]: Descarga una rama específica del servidor remoto.
    • Git pull --verbose [servidor_remoto]: Muestra un proceso detallado de la fusión.
    • Git pull --rebase [servidor_remoto]: Reordena los cambios locales sobre los remotos.

    El uso del comando git pull asegura que tu trabajo local esté actualizado con los cambios más recientes del repositorio remoto, evitando que trabajes con una versión desactualizada.

    Un error que he tenido fue que olvidé hacer pull antes de realizar cambios locales.

    3. Recuperación de cambios remotos.

    Si deseas ver los cambios remotos sin fusionarlos inmediatamente, puedes utilizar los siguientes comandos:

    • Git fetch [nombre_servidor_remoto]: Descarga los cambios del remoto sin fusionarlos.
    • Git checkout fetch_HEAD: Cambia a la rama que contiene los cambios descargados.
    • Git checkout -b [new_Branch]: Crea una nueva rama basada en los cambios descargados del remoto.
    • Git diff [rama_local] [rama_remota_cambios]: Compara diferencias entre una rama local y una remota.
    • Git merge [rama]: Fusiona los cambios de una rama con la actual.

    Utilizar git fetch te permite revisar los cambios disponibles en el repositorio remoto antes de decidir si los fusionas o los rebasas. Esto es útil para evitar sorpresas en tu código local.

    Los errores comunes incluyen olvidar realizar fetch regularmente, lo que lleva a trabajar con ramas desactualizadas, o confundir fetch con pull, que sí fusiona los cambios.

    4.Flujos de trabajo con GIT.

    Los flujos de trabajo en Git organizan la forma en que los equipos colaboran, utilizando ramas, pull requests, y estrategias como merge o rebase para gestionar los cambios. Dependiendo del proyecto, puedes optar por diferentes estrategias para mantener un desarrollo fluido y eficiente.

    5. Añadir colaboradores en GitHub.

    En GitHub, puedes añadir colaboradores a tu repositorio para que puedan contribuir al código. Es importante gestionar adecuadamente los permisos de los colaboradores para garantizar la seguridad del repositorio.

    Errores comunes incluyen no configurar correctamente los permisos, lo que puede permitir accesos no deseados o limitar a los colaboradores en su capacidad de contribuir.
